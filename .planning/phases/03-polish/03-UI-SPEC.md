# Phase 3 -- UI Design Contract

> Visual and interaction contract for the polish phase: cancel, conflict resolution, cross-platform. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Note:** This is a terminal UI (TUI) application built on tview/tcell. All visual specifications are expressed as tcell 256-color codes, character-based layout, and tview component styling. This spec extends the Phase 1/2 design system -- only changes and additions are documented here.

## Layout -- TransferModal Mode System

Source: D-03, D-08, RESEARCH.md Pattern 2

TransferModal introduces a multi-mode rendering system. The modal uses the same `*tview.Box` with `SetBorderPadding(2, 2, 5, 5)` (existing), switching display content based on mode enum.

### Mode States

```go
type modalMode int
const (
    modeProgress      modalMode = iota // Normal progress (existing)
    modeCancelConfirm                   // Cancel confirmation (new)
    modeConflictDialog                  // Conflict resolution (new)
    modeSummary                         // Transfer complete (existing)
)
```

### Mode 1: Progress (existing, unchanged)

```
+-- Uploading filename.txt ------------+
|                                       |
|     Uploading: filename.txt           |  <- Color255 (bright white)
|                                       |
|   ████████████░░░░░░░░  67%           |  <- Color248 (progress bar + text)
|                2.3 MB/s               |  <- Color245 (secondary)
|                ETA: 0m 12s            |  <- Color245 (secondary)
|                                       |
+---------------------------------------+
```

Source: existing `transfer_modal.go` Draw().

### Mode 2: Cancel Confirm (new)

Source: D-03 (first Esc = confirm prompt)

```
+-- Uploading filename.txt ------------+
|                                       |
|                                       |
|  Cancel transfer?                     |  <- #FFD700 (gold) -- warning, not error
|  [y] Yes  [n] No                     |  <- Color255 (options), Color245 (hints)
|                                       |
|                                       |
+---------------------------------------+
```

**Layout rules:**
- File name title unchanged from progress mode (shows what's being canceled)
- All progress content (bar, speed, ETA) hidden -- replaced by centered prompt
- Prompt text vertically centered within the modal inner area
- `[y]` and `[n]` displayed as tview-style tag hints (bracket letters in contrasting color)
- Footer hint: `"Press Esc to continue transfer"` in Color245

**Interaction:**
- `y` or `Enter` -- confirm cancel, call `cancel()`, mode -> modeSummary with canceled text
- `n` or `Esc` -- resume transfer, mode -> modeProgress (restore previous progress state)

### Mode 3: Conflict Dialog (new)

Source: D-06, D-07, D-08

```
+-- Uploading filename.txt ------------+
|                                       |
|  File already exists:                 |  <- #FFA500 (orange) -- warning
|  filename.txt (1.2M, 2024-03-15)     |  <- Color255 (existing file info)
|                                       |
|  [o] Overwrite  [s] Skip  [r] Rename  |  <- Color255 (options), Color245 (hints)
|                                       |
+---------------------------------------+
```

**Layout rules:**
- File name title unchanged from progress mode
- All progress content (bar, speed, ETA) hidden -- replaced by conflict prompt
- Warning line "File already exists:" in #FFA500 (orange)
- Existing file info line showing: `{filename} ({size}, {date})` in Color255
- Three options displayed as tview-style tag hints on a single line
- Options centered horizontally within the modal inner area

**Interaction (only these keys processed, all others ignored):**
- `o` -- Overwrite: send ConflictOverwrite on actionCh, mode -> modeProgress
- `s` -- Skip: send ConflictSkip on actionCh, mode -> modeProgress (continue to next file)
- `r` -- Rename: compute new name with `.1` suffix, send ConflictRename with new path on actionCh, mode -> modeProgress

**Multi-file conflict behavior (D-09):**
- Directory transfers show conflict dialog for each conflicting file individually
- Modal title updates to show file index: `" Uploading (file 3/10) "`
- After user resolves one conflict, transfer continues and next conflict triggers dialog again

### Mode 4: Summary -- Canceled (new variant)

Source: D-04

```
+-- Transfer Complete ------------------+
|                                       |
|  Transfer canceled                    |  <- #FF6B6B (red) -- destructive action completed
|  Cleaned up partial file              |  <- Color245 (secondary)

- Footer: existing `"Press any key to close"` hint unchanged

When directory transfer has skipped files due to conflict:

```
+-- Transfer Complete ------------------+

## Keyboard Interaction Map -- Phase 3 Additions

Source: D-03, D-07, D-08

These additions modify existing key behaviors. All Phase 1/2 keybindings remain unchanged in their original contexts.

| Key | Context | Action | Source |
|-----|---------|--------|--------|
| `Esc` | TransferModal visible (progress mode) | Switch to cancel confirm mode (modeCancelConfirm) | D-03 |
| `Esc` | TransferModal visible (cancel confirm mode) | Resume transfer, return to progress mode | D-03 |
| `y` | TransferModal visible (cancel confirm mode) | Confirm cancel: call `cancel()`, clean up partial file, show canceled summary | D-03, D-04 |
| `Enter` | TransferModal visible (cancel confirm mode) | Same as `y` -- confirm cancel | D-03 |
| `n` | TransferModal visible (cancel confirm mode) | Resume transfer, return to progress mode | D-03 |
| `o` | TransferModal visible (conflict dialog mode) | Overwrite: send ConflictOverwrite, resume transfer | D-07 |
| `s` | TransferModal visible (conflict dialog mode) | Skip: send ConflictSkip, continue to next file | D-07 |
| `r` | TransferModal visible (conflict dialog mode) | Rename: compute `file.1.txt` name, send ConflictRename, resume transfer | D-07 |

**Esc handler modification (D-03 -- double-Esc pattern):**

```
FileBrowser.handleGlobalKeys:
  Esc pressed:
    if transferModal.IsVisible():
      if transferModal.InCancelConfirm():
        // Second Esc in confirm mode -> resume transfer
        transferModal.ResumeProgress()
      else:
        // First Esc in progress/conflict mode -> enter cancel confirm
        transferModal.ShowCancelConfirm()
      return nil  // consumed
    else:
      close()     // existing behavior
```

**Event propagation in modal modes:**
- In `modeCancelConfirm`: TransferModal.HandleKey consumes `y`, `n`, `Enter`, `Esc`. All other keys ignored (return nil).
- In `modeConflictDialog`: TransferModal.HandleKey consumes `o`, `s`, `r`. All other keys ignored (return nil).
- In `modeProgress`: TransferModal.HandleKey consumes `Esc`. All other keys passed through.
- In `modeSummary`: TransferModal.HandleKey consumes all keys (any key dismisses).

Source: RESEARCH.md Pitfall 2 (key event handling), Pitfall 3 (goroutine synchronization)

## Rename Suffix Format

Source: D-07, RESEARCH.md Open Question 2 (recommendation: `file.1.txt`)

**Format:** `{stem}.{counter}{extension}`

Examples:
- `report.pdf` -> `report.1.pdf` -> `report.2.pdf`
- `photo.jpg` -> `photo.1.jpg`
- `notes.txt` -> `notes.1.txt`
- `Makefile` (no extension) -> `Makefile.1`

**Implementation:** `nextAvailableName()` function tries counters 1 through 100. Counter inserted between stem and extension using `filepath.Ext()` for local paths and custom split for remote paths (SFTP Unix paths).

## Goroutine Synchronization Contract

Source: RESEARCH.md Pattern 2, Pitfall 3

The conflict dialog requires careful goroutine synchronization between the transfer goroutine and the UI thread.

```
Transfer goroutine:                 UI thread (main):
  1. Stat() detects conflict
  2. Create actionCh (buffered 1)
  3. QueueUpdateDraw(func() {
       transferModal.ShowConflict(   4. Draw conflict dialog
         fileName, size, date,        5. User presses o/s/r
         actionCh)                  6. HandleKey sends action
     })                                on actionCh
  7. action := <-actionCh            (goroutine unblocks)
  8. Execute chosen action
  9. Continue transfer
```

**Key rules:**
- Transfer goroutine blocks on `<-actionCh` (correct -- it's in background, doesn't block UI)
- UI thread NEVER blocks on channel reads
- `actionCh` is buffered (capacity 1) to prevent goroutine leak if UI thread sends before goroutine reads
- `QueueUpdateDraw()` is used for all UI state changes from goroutines

## Cancel Cleanup Contract

Source: D-04, D-05, RESEARCH.md Pattern 1

When user confirms cancel:

1. `cancel()` called -> `ctx.Done()` channel closed
2. `copyWithProgress()` detects `ctx.Done()` at next chunk boundary (max 32KB delay)
3. `copyWithProgress()` returns `context.Canceled`
4. Caller (UploadFile/DownloadFile) closes file handles
5. Caller calls `sftp.Remove()` or `os.Remove()` to delete partial file
6. If cleanup fails: log `Warnw` with path and error, don't retry
7. `QueueUpdateDraw()` switches modal to canceled summary mode

**What is cleaned up:**
- Partial file on destination side -- always deleted (D-04)
- Partial file on source side -- never touched (read-only)
- SFTP connection -- NOT closed (D-05, reusable)
- Empty directories created during dir transfer -- NOT deleted (safe, no data cost)
