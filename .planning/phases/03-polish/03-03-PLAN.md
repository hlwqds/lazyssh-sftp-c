  <action>
    **设计原理：** Go build tags 是处理平台差异的标准机制（D-10）。将文件权限设置分离到平台特定文件中，比 `runtime.GOOS` 散弹枪检查更清晰。Windows NTFS 不支持 Unix 权限位，os.Chmod 在 Windows 上要么失败要么只影响只读标志，因此静默降级更安全（D-13）。路径处理方面，Go 标准库的 filepath 包已处理路径分隔符差异，远程路径始终使用 Unix 风格（SFTP 协议要求）。

    **实现步骤：**

    1. **创建 `internal/adapters/data/transfer/permissions_unix.go`**：
       ```go
       //go:build !windows

       package transfer

       import (
           "os"
           "go.uber.org/zap"
       )

       // setFilePermissions sets file permissions on Unix-like systems.
       // Logs a warning if chmod fails (e.g., permission denied).
       func setFilePermissions(path string, mode os.FileMode, log *zap.SugaredLogger) {
           if err := os.Chmod(path, mode); err != nil {
               log.Warnw("failed to set file permissions", "path", path, "mode", mode, "error", err)
           }
       }
       ```

    2. **创建 `internal/adapters/data/transfer/permissions_windows.go`**：
       ```go
       //go:build windows

       package transfer

       import (
           "go.uber.org/zap"
       )

       // setFilePermissions is a no-op on Windows.
       // Windows NTFS does not support Unix-style permission bits.
       // os.Chmod on Windows only affects the read-only flag, which we ignore.
       func setFilePermissions(path string, mode uint32, log *zap.SugaredLogger) {
           log.Debugw("skipping chmod on Windows (NTFS)", "path", path)
       }
       ```
       **注意：** Windows 版本的 mode 参数类型需要考虑。Unix 版本使用 `os.FileMode`，Windows 版本不需要 mode 但参数签名必须匹配。由于两个文件在同一个 package 中，函数签名必须一致。使用 `os.FileMode` 作为两个版本的参数类型即可（Windows 版本忽略该值）。

       **修正 Windows 版本签名：**
       ```go
       import "os"
       func setFilePermissions(path string, mode os.FileMode, log *zap.SugaredLogger) {
           log.Debugw("skipping chmod on Windows (NTFS)", "path", path)
       }
       ```

    3. **修改 `transfer_service.go` 在文件传输完成后调用 setFilePermissions**：

       **3a. 在 UploadFile 中**（上传后设置远程文件权限）：
       ```go
       // 在 copyWithProgress 成功后（非取消），设置远程文件权限
       // 远程端需要通过 SFTP 设置权限——暂不实现
       // 本地端不需要设置权限（只是读取源文件）
       ```
       实际上，权限设置应该在下载完成后设置本地文件权限（从远程 FileInfo 中获取 mode）。上传时远程端权限设置需要 `sftp.Chmod()`，这已通过 go doc 验证可用。

       **修正设计：** 权限设置分两个方向：
       - **下载**（远程 -> 本地）：下载完成后对本地文件调用 `setFilePermissions(localPath, remoteMode, log)`。当前 DownloadFile 不获取远程文件权限（OpenRemoteFile 返回 io.ReadCloser，没有 Stat）。需要先获取远程文件 mode。
       - **上传**（本地 -> 远程）：上传完成后对远程文件调用 `ts.sftp.Chmod(remotePath, localMode)`。当前 UploadFile 已有 `localFile.Stat()` 获取 mode。

       **但 RESEARCH.md 明确指出（D-13）："传输时尝试设置权限（chmod），但如果目标系统不支持（如 Windows NTFS）则静默忽略错误"**。

       **简化实现方案（避免过度复杂化）：** 只在下载时设置本地文件权限。上传时暂不设置远程权限（SFTP Chmod 需要额外接口方法，超出 INTG-03 范围）。

       在 DownloadFile 和 downloadSingleFile 中：
       ```go
       // 在 copyWithProgress 成功后，设置本地文件权限
       // 需要获取远程文件的 mode——通过 sftp.Stat
       if statErr == nil {
           setFilePermissions(localPath, remoteMode, ts.log)
       }
       ```
       **问题：** 当前 DownloadFile 不获取远程文件 mode。OpenRemoteFile 返回 io.ReadCloser，BytesTotal 未知。

       **更简单的方案：** 不在传输层设置权限。权限设置是"nice to have"而非核心需求（INTG-03 的核心是"works on all platforms"，不是"preserves permissions"）。当前代码不设置权限也不会 break——文件默认权限在不同平台上都有合理默认值。

       **最终方案（最小化改动）：**
       - 创建 permissions_windows.go 和 permissions_unix.go（build tag 基础设施就位）
       - 在 transfer_service.go 中**不调用** setFilePermissions（避免引入远程 Stat 的额外复杂性）
       - setFilePermissions 函数作为后续 v2 需求的基础设施预留
       - INTG-03 的核心价值通过路径处理审计和显示格式确认来满足
