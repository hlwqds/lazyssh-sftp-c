## Spacing Scale

This is a terminal UI. Spacing is character-cell-based, not pixel-based. The following cell-level spacing conventions are established in the codebase (unchanged from Phase 7):

| Token | Cells | Usage |
|-------|-------|-------|
| Compact | 0 | Inline prefixes (`[C] `, `[M] `, `* `) |
| Standard | 1 | Cell padding in Table columns |
| Section | 1 row | Separator rows in overlay dialogs |
| Status | 1 row | Status bar (fixed height at bottom) |
| Modal padding | 2,2,5,5 | TransferModal Box border padding (top, bottom, left, right) |

Exceptions: ConfirmDialog uses 60% terminal width (max 70 columns), 7-8 rows centered. Conflict dialog within TransferModal uses existing drawConflictDialog layout. No new spacing exceptions for Phase 8.

## Typography

Terminal UI uses the user's configured monospace font. Style variants are tcell Style attributes (bold, color). All values from Phase 7 remain unchanged. New additions for Phase 8:

| Role | Style | tcell Color | Usage |
|------|-------|-------------|-------|
| Header | Bold | Color255 on Color235 | Table header row (unchanged) |
| Body file | Normal | Color252 | Regular file name in listing (unchanged) |
| Body directory | Normal | Color33 | Directory name in listing (unchanged) |
| Secondary | Normal | Color245 | Size, Modified, Permissions, footer hints (unchanged) |
| Title | Normal | Color250 | Pane titles, dialog titles (unchanged) |
| Status key hint | Normal | `white` (tview tag) | Keyboard shortcut labels in status bar (unchanged) |
| Error | Normal | #FF6B6B | Status bar error messages, connection failures (unchanged) |
| Warning | Normal | #FFD700 | Cancel confirmation, Space multi-select prefix (unchanged) |
| Conflict | Normal | #FFA500 | File conflict dialog text (unchanged) |
| Clipboard copy marker | Normal | #00FF7F | `[C]` prefix for copy-marked files (Phase 7, unchanged) |
| **Move marker (NEW)** | **Normal** | **#FF6B6B** | **`[M]` prefix for move-marked files (MOV-01)** |
| Success | Normal | #A0FFA0 | Connection success message (unchanged) |

Weight constraint: Bold is used only for table header row. All other text is normal weight. The `[M]` prefix row uses `tcell.AttrBold` on the name cell (same pattern as `[C]` in Phase 7, per local_pane.go:178 and remote_pane.go:233).

**Move marker color rationale:** #FF6B6B (red) is chosen to visually distinguish move from copy (#00FF7F green). Red communicates "destructive intent" -- the source file will be deleted after the move. This aligns with the existing Destructive color token already used for errors and cancel-confirmed text. Source: CONTEXT.md Claude's Discretion -- "suggest with distinction from [C], e.g. yellow/red".

## Interaction Contract

### Key Bindings

| Key | Context | Action | Feedback |
|-----|---------|--------|----------|
| `x` | No overlay visible, pane focused | Mark current file as move source | Status bar: "Move: {filename}" in red; file row shows `[M]` prefix |
| `p` | No overlay visible, pane focused, clipboard active (OpMove) | Move file to current directory (copy + delete source) | Local: instant (status bar flash). Remote: TransferModal shows move progress |
| `p` | No overlay visible, clipboard active (OpCopy) | Copy file to current directory | (unchanged from Phase 7) |
| `p` | No overlay visible, clipboard empty | No action | Silent (no feedback) |
| `p` | No overlay visible, clipboard active, different pane | Reject cross-pane paste | Status bar error: "Cross-pane paste not supported (v1.3+)" |
| `Esc` | No overlay visible, clipboard active | Clear clipboard | Status bar: "Clipboard cleared" in green; prefix removed from file |
| `c` | No overlay visible, clipboard already active (OpMove) | Replace move clipboard with copy clipboard | Old clipboard cleared, new file marked with `[C]` |
| `x` | No overlay visible, clipboard already active (OpCopy) | Replace copy clipboard with move clipboard | Old clipboard cleared, new file marked with `[M]` |

### Esc Priority Chain (unchanged from Phase 7)

1. InputDialog visible -> delegate to InputDialog.HandleKey
2. ConfirmDialog visible -> delegate to ConfirmDialog.HandleKey
3. RecentDirs visible -> delegate to RecentDirs.HandleKey
4. TransferModal visible -> delegate to TransferModal.HandleKey
5. Clipboard active -> clear clipboard, return nil (do NOT close browser)
6. Otherwise -> close file browser

### Clipboard State Machine (extended from Phase 7)

```
Empty --[c]--> Active(OpCopy, sourcePane, fileInfo, sourceDir)
Empty --[x]--> Active(OpMove, sourcePane, fileInfo, sourceDir)     // NEW Phase 8
Active(OpCopy) --[c]--> Active(OpCopy, newSource...)               // replace
Active(OpCopy) --[x]--> Active(OpMove, newSource...)               // NEW: replace
Active(OpMove) --[c]--> Active(OpCopy, newSource...)               // NEW: replace
Active(OpMove) --[x]--> Active(OpMove, newSource...)               // NEW: replace
Active(OpCopy) --[p, success]--> Empty                             // auto-clear (D-07)
Active(OpMove) --[p, success]--> Empty                             // auto-clear (D-07)
Active(OpCopy) --[p, failure]--> Active(OpCopy)                    // keep for retry (D-07)
Active(OpMove) --[p, failure]--> Active(OpMove)                    // keep for retry (D-07)
Active(any) --[Esc]--> Empty                                       // explicit clear (CLP-03)
```

### [M] Prefix Rendering Rules

Location: Name column (column 0) in LocalPane/RemotePane populateTable.

Rendering logic (extends Phase 7 clipboardProvider):

1. If clipboard is active AND clipboard.FileInfo.Name matches current file AND clipboard.SourceDir matches current pane path:
   - If `clipboard.Operation == OpMove`:
     - Prepend `[M] ` to nameText
     - Set nameColor to `#FF6B6B` (red)
     - Set nameBg to `tcell.Color236` (dark background for visibility)
     - Set nameAttrs to `tcell.AttrBold`
   - Else (OpCopy):
     - Prepend `[C] ` to nameText
     - Set nameColor to `#00FF7F` (green)
     - Set nameBg to `tcell.Color236`
     - Set nameAttrs to `tcell.AttrBold`
2. Else if file is in `selected` map:
   - Prepend `* ` to nameText (existing behavior)
   - Set nameColor to `#FFD700` (gold)
3. Otherwise: normal rendering.

Priority: clipboard `[M]`/`[C]` takes precedence over Space `* ` selection.

### TransferModal modeMove

New mode value added to the existing `modalMode` enum:

```go
const (
    modeProgress       modalMode = iota
    modeCancelConfirm
    modeConflictDialog
    modeSummary
    modeCopy           // Phase 7: remote copy progress
    modeMove           // Phase 8: remote move progress
)
```

**Draw behavior:** modeMove reuses `drawProgress()` -- identical layout to modeCopy and modeProgress (file label + progress bar + speed + ETA). Draw switch case: `case modeProgress, modeCopy, modeMove:`.

**HandleKey behavior:** modeMove shares the same HandleKey case: `case modeProgress, modeCopy, modeMove:`. Esc triggers cancel confirmation, all other keys pass through.

**ShowMove method:**
```go
func (tm *TransferModal) ShowMove(filename string) {
    tm.visible = true
    tm.mode = modeMove
    tm.cancelConfirmed = false
    tm.SetTitle(fmt.Sprintf(" Moving %s ", filename))
    tm.bar = NewProgressBar()
    tm.speedSamples = tm.speedSamples[:0]
    tm.fileLabel = fmt.Sprintf("Moving: %s", filename)
    tm.infoLine = ""
    tm.etaLine = ""
}
```

**"Deleting source..." phase:** After the copy phase completes successfully, before deleting the source file, update the TransferModal display fields directly via QueueUpdateDraw:

```go
fb.app.QueueUpdateDraw(func() {
    fb.transferModal.fileLabel = "Deleting source..."
    fb.transferModal.infoLine = ""
    fb.transferModal.etaLine = ""
})
```

This reuses the existing drawProgress row layout (row 1 = fileLabel, row 4 = infoLine, row 5 = etaLine). No new mode or draw method is needed for this transient phase. The progress bar and speed/ETA lines are cleared, leaving only the "Deleting source..." text centered at row 1. Source: RESEARCH.md Pattern 3 -- "suggested not to add new mode".

**Summary:** On completion, calls `ShowSummary()` with appropriate counts. On copy-phase failure, calls `ShowSummary()` with failure info. On delete-source failure (partial failure per D-04), hides modal and shows status bar error.

### Conflict Dialog Integration (D-01)

**All paste operations (copy AND move) show conflict dialog when target file exists.** This replaces the Phase 7 same-directory auto-rename behavior.

**Implementation:** In `handlePaste()`, replace the `nextAvailableName()` auto-rename block with a conflict check + `buildConflictHandler()` flow:

```
1. Compute targetPath
2. Stat(targetPath) to check if target exists
3. If exists:
   a. Create context with cancel
   b. Call buildConflictHandler(ctx)
   c. Call onConflict(targetName) -- blocks until user responds
   d. Handle result: Skip (return), Rename (update targetPath), Overwrite (continue)
4. If not exists: proceed normally
```

**Conflict dialog layout:** Reuse existing `TransferModal.ShowConflict()` and `drawConflictDialog()`. No layout changes.

**Default selection:** The conflict dialog does not have a highlighted default -- it waits for explicit key input (`o`/`s`/`r`/Esc). This is unchanged from Phase 3. Source: CONTEXT.md Claude's Discretion -- "suggest Skip or Rename, not Overwrite" -- resolved by keeping the existing no-default behavior where Esc maps to Skip.

**Local paste conflict:** Local paste operations must also go through the conflict dialog. This means `handleLocalPaste()` must be refactored to run in a goroutine (same pattern as `handleRemotePaste()`), so the channel-based buildConflictHandler can block without freezing the UI. Source: RESEARCH.md Open Question 3.

### Same-Directory Move Optimization

When the user marks a file with `x` and presses `p` in the **same directory**, the move is equivalent to a rename. This is handled as a special case in `handlePaste()`:

```
if clipboard.Operation == OpMove && currentPath == clipboard.SourceDir:
    // Check for name conflict via conflict dialog
    // If no conflict or user chose overwrite:
    go func() {
        err := fileService.Rename(sourcePath, targetPath)
        QueueUpdateDraw { handle success/failure }
    }()
    return  // skip Copy+Delete flow
```

This is more efficient (atomic rename) and avoids the copy+delete failure scenario for same-directory moves. Source: RESEARCH.md Pitfall 2.

### Remote Move -- Two-Phase Flow (D-03, D-04, D-08)

Phase 1 -- Copy: Uses `CopyRemoteFile`/`CopyRemoteDir` with TransferModal in modeMove. Shows "Moving: {filename}" title and "Downloading: {filename}" / "Uploading: {filename}" phase labels. Progress bar, speed, ETA all active.

Phase 2 -- Delete source: After copy succeeds, update TransferModal to show "Deleting source..." (fileLabel), clear infoLine and etaLine. Execute `SFTPService.Remove()` or `SFTPService.RemoveAll()`. This is a single SFTP RPC -- instantaneous, no progress bar needed.

**Failure handling (D-04):**
- Copy fails: Source preserved. Show status error. Clipboard NOT cleared (allow retry). Hide modal.
- Copy succeeds, delete fails: Attempt cleanup (delete target copy). If cleanup succeeds: "Move failed: source kept, target cleaned up". If cleanup fails: "Move failed: source kept, target copy may need manual cleanup". Clipboard NOT cleared. Hide modal.

## File Operations Handler Pattern

### handleMove (new)

Follows the established handler pattern (mirrors handleCopy with OpMove):

1. Get active selection row and cell
2. Extract FileInfo reference
3. Validate: row >= 1, cell not nil, reference is FileInfo
4. For remote pane: check `IsConnected()`, show error if not
5. Create Clipboard struct: `Active=true, SourcePane=fb.activePane, FileInfo=fi, SourceDir=currentPath, Operation=OpMove`
6. Call `fb.refreshPane(fb.activePane)` to trigger `[M]` prefix rendering
7. Call `fb.focusOnItem(fb.activePane, fi.Name)` to restore cursor
8. Call `fb.updateStatusBarTemp(fmt.Sprintf("[#FF6B6B]Move: %s[-]", fi.Name))`

### handlePaste (modified)

Extended from Phase 7. New dispatch logic:

1. Guard: `if !fb.clipboard.Active { return }` (silent)
2. Guard: cross-pane check (unchanged)
3. Guard: remote connection check (unchanged)
4. Compute source and target paths
5. **Conflict check (NEW):** Stat targetPath, if exists -> buildConflictHandler -> handle result (Skip/Rename/Overwrite)
6. **Operation dispatch (NEW):**
   - If `clipboard.Operation == OpMove && currentPath == clipboard.SourceDir`: same-directory move via `Rename()`
   - If `clipboard.Operation == OpMove`: `handleLocalMove()` or `handleRemoteMove()`
   - If `clipboard.Operation == OpCopy`: `handleLocalPaste()` or `handleRemotePaste()` (existing)
7. On success: clear clipboard (D-07), refresh pane, position cursor
8. On failure: show status error, do NOT clear clipboard (D-07)

### handleLocalMove (new)

Same-directory optimization:
```go
go func() {
    err := fs.Rename(sourcePath, targetPath)
    fb.app.QueueUpdateDraw(func() {
        if err != nil {
            fb.showStatusError(fmt.Sprintf("Move failed: %s", trimError(err.Error(), 50)))
            return
        }
        fb.clipboard = Clipboard{}
        fb.refreshPane(fb.activePane)
        fb.focusOnItem(fb.activePane, targetName)
        fb.updateStatusBarTemp(fmt.Sprintf("[#FF6B6B]Moved: %s[-]", targetName))
    })
}()
```

Note: Same-directory move uses Rename (atomic). Cross-directory move falls through to Copy+Delete. Source: RESEARCH.md Pitfall 2.

### handleRemoteMove (new)

Two-phase goroutine with cleanup:

```go
// Phase 1: Copy via CopyRemoteFile/CopyRemoteDir (modeMove progress)
// Phase 2: Delete source via SFTPService.Remove/RemoveAll
// On Phase 2 failure: attempt cleanup of target copy (D-04)
```
